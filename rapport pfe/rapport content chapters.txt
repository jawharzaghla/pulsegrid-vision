Chapitre II : Étude théorique

Section 1 : La Business Intelligence et les tableaux de bord
1.1 Définition et évolution
La Business Intelligence (BI) regroupe les technologies et les méthodes qui transforment des données brutes en informations exploitables pour l'aide à la décision. Popularisé en 1989, ce concept visait initialement à améliorer la prise de décision par des faits vérifiables.
L'évolution de la BI se divise en trois grandes étapes :
•	Années 1990 : Utilisation de rapports statiques et d'entrepôts de données gérés exclusivement par les services informatiques.
•	Années 2000 : Apparition du traitement analytique en ligne (OLAP) pour explorer les données sous plusieurs dimensions.
•	Années 2010 : Avènement de la BI en libre-service avec des outils permettant aux utilisateurs non techniques de créer leurs propres analyses.
1.2 Architecture et niveaux de traitement
Le système PulseGrid repose sur trois niveaux fondamentaux de gestion de l'information:
Niveau	Rôle	Application PulseGrid
Collecte	Extraction des données sources	Connexion via REST API 
Traitement	Nettoyage et normalisation	DataMapping et CleanedMetricPayload 
Présentation	Visualisation graphique	Canevas Dashboard avec ApexCharts 
1.3 Typologies et indicateurs (KPI)
La littérature sépare la BI en trois catégories principales :
1.	BI Opérationnelle : Fournit des données en temps réel pour les décisions quotidiennes. PulseGrid s'inscrit dans ce modèle par ses widgets rafraîchissables.
2.	BI Stratégique : Analyse les tendances sur le long terme pour les dirigeants.
3.	BI Collaborative : Partage des analyses au sein d'une équipe, géré dans PulseGrid par le système de projets multi-utilisateurs.
Pour mesurer cette performance, on utilise des indicateurs clés (KPI) qui doivent respecter les critères SMART : Spécifiques, Mesurables, Atteignables, Réalistes et Temporels.

4.	Lors de la fermeture de la session, la clé en mémoire est supprimée (zéroïsation).

Section 3 : Technologies et Développement Frontend
3.1 Framework Angular et structuration
Angular est un framework structuré qui impose une architecture par composants autonomes. Le projet utilise plusieurs mécaniques avancées :
•	Injection de dépendances : Centralise les services comme l'authentification ou le traitement des API pour qu'ils soient uniques dans toute l'application.
•	Lazy Loading : Charge les modules uniquement lorsqu'ils sont nécessaires, ce qui optimise la vitesse d'affichage initiale.
•	Intercepteurs HTTP : Ajoute automatiquement les jetons de sécurité aux requêtes et gère les erreurs de connexion de manière centralisée.














 
Interactions entre les couches de l'architecture Angular

3.2 Outillage technique
Le développement repose sur un ensemble d'outils complémentaires :
1.	TypeScript : Assure la détection des erreurs dès la phase de programmation en imposant un typage strict des données.
2.	Tailwind CSS : Permet une gestion visuelle par classes utilitaires, évitant les conflits de styles entre les composants.
3.	ApexCharts : Fournit la bibliothèque graphique pour transformer les résultats JSON en visualisations interactives (courbes, barres, jauges).
3.3 Intelligence Artificielle et analyse de données
L'innovation de la plateforme réside dans l'usage de modèles de langage (LLM) pour l'analyse de données structurées. PulseGrid utilise l'infrastructure Groq pour obtenir des analyses rapides.
Scénario de traitement d'une analyse IA :
1.	Le système extrait les métriques normalisées (CleanedMetricPayload) sans inclure de données brutes.
2.	Un prompt structuré est envoyé au modèle incluant le contexte et les unités de mesure.
3.	Le résultat est retourné sous format Markdown progressif (streaming) pour une lecture immédiate par l'utilisateur.

Résumé
Ce rapport présente la conception et le développement de PulseGrid, une plateforme SaaS de Business Intelligence destinée aux opérateurs multi-entreprises. Dans un contexte où les dirigeants gèrent simultanément plusieurs activités commerciales, chacune disposant de ses propres tableaux de bord, outils de reporting et interfaces de connexion, la fragmentation de l'information constitue un frein majeur à la prise de décision efficace.

PulseGrid répond à cette problématique en centralisant l'ensemble des indicateurs clés de performance (KPI) dans un espace de travail unifié. L'architecture retenue est volontairement légère : aucun backend de stockage de données n'est requis, les données étant récupérées en temps réel directement depuis des API REST tierces. Les identifiants de connexion sont protégés par chiffrement AES-GCM 256 bits via l'API Web Crypto du navigateur.

La plateforme intègre un moteur d'analyse basé sur l'intelligence artificielle, exploitant le modèle Llama 3.3 70B via l'API Groq, pour fournir des analyses automatisées, détecter les anomalies et générer des rapports synthétiques en langage naturel.

Sur le plan technique, le frontend est développé avec Angular et Tailwind CSS, les visualisations sont réalisées avec ApexCharts, et l'ensemble de l'architecture respecte les principes SOLID, le pattern feature-based et les bonnes pratiques Angular (Signals, lazy loading, intercepteurs HTTP).

Mots-clés : Business Intelligence, SaaS, Angular, REST API, Web Crypto, Groq, LLM, Dashboard, Multi-entreprises, KPI.




Abstract
This report presents the design and development of PulseGrid, a SaaS Business Intelligence platform targeting multi-business operators. Information fragmentation across multiple ventures severely hampers effective decision-making.

PulseGrid centralizes all Key Performance Indicators into a single workspace. No data storage backend is required — data is fetched in real time from third-party REST APIs. Credentials are protected through AES-GCM 256-bit encryption via the browser's Web Crypto API.

The platform features an AI analysis engine leveraging Llama 3.3 70B via the Groq API for automated metric analysis, anomaly detection, and natural-language summaries. The frontend is built with Angular, Tailwind CSS, and ApexCharts, following SOLID principles and Angular best practices.

Keywords: Business Intelligence, SaaS, Angular, REST API, Web Crypto, Groq, LLM, Dashboard, Multi-business, KPI.

Introduction Générale
Contexte général
L'essor de l'entrepreneuriat numérique et la multiplication des modèles économiques basés sur les données ont engendré une nouvelle réalité pour les opérateurs multi-entreprises : la nécessité de piloter simultanément plusieurs activités commerciales, chacune générant des flux de données hétérogènes, stockés dans des outils distincts et accessibles via des interfaces différentes.

Cette fragmentation de l'information crée un coût opérationnel invisible mais significatif. Selon plusieurs études sur la productivité des dirigeants, la navigation entre différents outils de reporting peut absorber entre 45 minutes et 2 heures par jour — un temps précieux soustrait à la prise de décision stratégique.

Par ailleurs, l'avènement des modèles de langage de grande taille (LLM) et des API d'inférence ultra-rapides comme Groq ouvre une nouvelle perspective : celle d'un assistant analytique intégré, capable d'interpréter des données métier complexes et de les restituer en langage naturel, en quelques secondes.

Motivation du projet
PulseGrid naît d'un besoin concret et vécu : un opérateur gérant plusieurs entreprises simultanément — boutique e-commerce, application SaaS, portefeuille immobilier, agence de marketing — se retrouve chaque matin à enchaîner les connexions, les chargements de pages, les exports de rapports, sans jamais avoir une vision globale cohérente.

L'idée centrale est radicale dans sa simplicité : connecter n'importe quelle API REST, configurer un widget en quelques clics, et disposer instantanément d'un tableau de bord intelligent, unifié et analysé par une IA. Sans base de données propre, sans entrepôt de données, sans infrastructure lourde.

Objectifs du projet
Ce projet de fin d'études vise à concevoir et développer PulseGrid, une plateforme SaaS de Business Intelligence répondant aux objectifs suivants :

1.	Centraliser les KPI de plusieurs entreprises dans un espace de travail unique et unifié
2.	Permettre la connexion à n'importe quelle API REST via un système de widgets configurables
3.	Garantir la sécurité des identifiants API par un chiffrement client-side AES-GCM via Web Crypto
4.	Intégrer une analyse IA en temps quasi-réel via le modèle Llama 3.3 70B (Groq)
5.	Offrir une expérience utilisateur premium et responsive via Angular et Tailwind CSS
6.	Respecter une architecture logicielle propre, maintenable et scalable

Contexte et Problématique

1.1 Présentation du cadre du projet
Ce projet s'inscrit dans le cadre d'un Projet de Fin d'Études en Génie Logiciel, orienté vers la conception et le développement d'une solution SaaS complète, de la spécification des besoins jusqu'au déploiement. PulseGrid est une application web de type SaaS (Software as a Service) qui s'adresse à un marché ciblé : les opérateurs d'entreprises multiples, les agences gérant plusieurs clients, les holdings et les indépendants disposant de plusieurs sources de revenus. Chacun de ces profils partage la même contrainte fondamentale : une vision éclatée de leurs données métier.

Définition – Business Intelligence (BI)
La Business Intelligence désigne l'ensemble des technologies, processus et outils permettant
la collecte, l'intégration, l'analyse et la présentation des données d'entreprise.
L'objectif est de transformer des données brutes en informations exploitables pour la prise
de décision stratégique et opérationnelle.














1.2 Analyse de la problématique
La problématique centrale peut se formuler ainsi : Comment permettre à un opérateur gérant plusieurs entités commerciales hétérogènes d'avoir une vision consolidée, en temps réel, de l'ensemble de ses indicateurs de performance, sans investissement en infrastructure de données ?

1.2.1 Le coût caché de la fragmentation
Une journée type d'un opérateur multi-entreprises ressemble à la séquence suivante : connexion au tableau de bord Shopify de la boutique A, export du rapport de ventes, connexion au CRM de l'entreprise B, vérification des métriques Google Analytics, ouverture de Stripe pour contrôler les revenus SaaS, puis répétition du cycle pour chaque entité supplémentaire.

Problème	Impact mesurable
Interfaces hétérogènes	Charge cognitive élevée, erreurs d'interprétation
Multiples connexions quotidiennes	45 à 90 minutes perdues par jour
Absence de vue consolidée	Décisions sans vision globale cohérente
Pas d'analyse croisée	Opportunités et corrélations invisibles
Pas d'alerte intelligente	Anomalies détectées trop tard


1.2.2 Les limites des solutions actuelles
Les outils BI existants comme Tableau, Power BI ou Looker répondent aux besoins des grandes entreprises mais imposent des contraintes rédhibitoires pour les profils ciblés par PulseGrid : nécessité d'un entrepôt de données (data warehouse), coûts élevés, configurations complexes et courbes d'apprentissage longues.

1.3 Solution proposée : PulseGrid
PulseGrid propose une approche fondamentalement différente : au lieu d'ingérer et de stocker les données, la plateforme les récupère à la demande depuis les API REST des outils métier existants, les normalise à la volée et les présente dans un tableau de bord unifié, configurable par widget.

Cette architecture « API-first, zéro stockage » présente plusieurs avantages décisifs :
•	Déploiement immédiat : aucune infrastructure de données à provisionner
•	Confidentialité renforcée : les données ne transitent jamais par les serveurs PulseGrid
•	Flexibilité maximale : tout service exposant une API REST peut être connecté
•	Coût maîtrisé : pas de base de données volumétrique ni de pipeline ETL

Chapitre 3
Conception de l'Architecture

3.1 Architecture générale du système
PulseGrid repose sur une architecture à deux niveaux distincts, séparant strictement les responsabilités selon un principe fondamental : aucune donnée métier ne transite par l'infrastructure PulseGrid.

3.1.1 Niveau 1 — Frontend Angular (client)
Le frontend constitue le cœur de la plateforme. Il est responsable de l'intégralité de la logique de récupération de données : les requêtes REST partent directement du navigateur de l'utilisateur vers les API tierces. Les réponses sont reçues, normalisées et visualisées dans le même contexte navigateur, sans jamais transiter par un serveur PulseGrid.

3.1.2 Niveau 2 — Backend léger (auth + proxy IA)
Un backend minimal (Node.js/Express ou Supabase) est maintenu pour deux responsabilités exclusives : la gestion des comptes utilisateurs et l'authentification JWT, et la proxification des appels à l'API Groq. Ce backend ne stocke, ne reçoit et ne voit jamais les données des API tierces des clients.

Principe architectural clé
La séparation entre le flux de données métier (direct navigateur → API tierce) et le flux
d'authentification/IA (navigateur → backend PulseGrid → Groq) est le fondement de la
proposition de valeur sécurité de PulseGrid. Les clés API des clients ne quittent jamais
leur navigateur en clair.

3.5 Architecture de sécurité
3.5.1 Chiffrement AES-GCM avec Web Crypto API
Le schéma de chiffrement repose exclusivement sur les primitives cryptographiques natives du navigateur, exposées via l'interface SubtleCrypto. Cette approche garantit que le code de chiffrement est exécuté dans un contexte sécurisé et auditable, sans dépendance externe.

Le processus complet, depuis la création du compte jusqu'au déchiffrement en session, suit les étapes suivantes :

7.	Génération d'un sel aléatoire de 16 octets via crypto.getRandomValues()
8.	Dérivation d'une clé AES-256 depuis le mot de passe utilisateur via PBKDF2 (100 000 itérations, SHA-256)
9.	Chiffrement de chaque clé API avec AES-GCM (IV aléatoire de 12 octets par opération)
10.	Stockage en localStorage : uniquement le sel et les données chiffrées (ciphertext + IV en base64)
11.	À la connexion : re-dérivation de la clé depuis le mot de passe, déchiffrement en mémoire uniquement
12.	À la déconnexion : zéro-isation de la clé dérivée depuis la mémoire

3.5.2 Gestion des tokens JWT
L'authentification JWT repose sur une double paire de tokens : un access token à courte durée de vie (15 minutes) transmis dans les en-têtes Authorization des requêtes API, et un refresh token à longue durée (7 jours) stocké dans un cookie httpOnly non accessible par JavaScript.

L'intercepteur HTTP Angular (AuthInterceptor) gère automatiquement le cycle de vie des tokens : injection de l'access token dans chaque requête sortante, détection des erreurs 401, tentative de renouvellement silencieux via le refresh token, et redirection vers la page de connexion si le renouvellement échoue.

3.5.3 Protection contre les attaques XSS
Angular offre une protection XSS native via son moteur de templates : toutes les valeurs interpolées dans les templates HTML sont automatiquement échappées. Cette protection est complétée par une politique de sécurité du contenu (Content Security Policy) configurée au niveau du serveur de déploiement.

3.6 Flux de données
3.6.1 Flux widget (REST → affichage)
Au chargement d'un tableau de bord, tous les widgets déclenchent leurs requêtes en parallèle via l'opérateur RxJS forkJoin. Chaque requête suit le pipeline suivant :
13.	Déchiffrement des credentials du widget (AES-GCM)
14.	Construction de la requête HTTP avec les en-têtes d'authentification appropriés
15.	Exécution de la requête vers l'API tierce
16.	Application du mapping JSON pour extraire les valeurs de métrique
17.	Émission du CleanedMetricPayload vers le composant widget-shell
18.	Rendu du composant de visualisation correspondant

L'échec d'un widget n'affecte jamais les autres : chaque Observable est isolé avec catchError, retournant un état d'erreur typé plutôt qu'une exception non gérée.

3.6.2 Flux IA (métriques → analyse → rapport)
L'analyse IA ne transmet jamais les réponses brutes des API tierces. Seuls les CleanedMetricPayload — les valeurs déjà extraites et normalisées — sont envoyés au backend proxy, qui les transmet à Groq avec un prompt système structuré demandant une analyse en sections : Résumé, Insights métriques et Anomalies détectées.

4.2.1 Service AuthService
Le service d'authentification gère l'intégralité du cycle de vie des sessions utilisateur. Il expose des méthodes observables pour la connexion, la déconnexion, le renouvellement de token et la vérification de l'état d'authentification.

// core/services/auth.service.ts
@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly currentUser = signal<User | null>(null);
  readonly isAuthenticated = computed(() => this.currentUser() !== null);

  login(email: string, password: string): Observable<User> {
    return this.http.post<AuthResponse>('/api/auth/login', { email, password })
      .pipe(
        tap(response => {
          this.storeAccessToken(response.accessToken);
          this.currentUser.set(response.user);
        }),
        map(response => response.user)
      );
  }

  refreshToken(): Observable<string> {
    return this.http.post<{ accessToken: string }>('/api/auth/refresh', {})
      .pipe(tap(r => this.storeAccessToken(r.accessToken)));
  }
}

4.2.2 Intercepteur HTTP
L'intercepteur HTTP est responsable de l'injection automatique du JWT dans les en-têtes de chaque requête sortante, et de la gestion transparente du renouvellement de token en cas d'expiration.

// core/interceptors/auth.interceptor.ts
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getAccessToken();

  const authReq = token
    ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
    : req;

  return next(authReq).pipe(
    catchError(error => {
      if (error.status === 401) {
        return authService.refreshToken().pipe(
          switchMap(newToken => {
            const retryReq = req.clone({
              setHeaders: { Authorization: `Bearer ${newToken}` }
            });
            return next(retryReq);
          }),
          catchError(() => {
            authService.logout();
            return throwError(() => error);
          })
        );
      }
      return throwError(() => error);
    })
  );
};

4.3 Service de chiffrement Web Crypto
Le CryptoService est la pièce maîtresse de la sécurité de PulseGrid. Il encapsule toutes les opérations cryptographiques en utilisant exclusivement l'API SubtleCrypto native du navigateur, garantissant l'absence de dépendances tierces dans la chaîne de chiffrement.

// core/services/crypto.service.ts
@Injectable({ providedIn: 'root' })
export class CryptoService {

  generateSalt(): Uint8Array {
    return crypto.getRandomValues(new Uint8Array(16));
  }

  async deriveKey(password: string, salt: Uint8Array): Promise<CryptoKey> {
    const keyMaterial = await crypto.subtle.importKey(
      'raw',
      new TextEncoder().encode(password),
      'PBKDF2',
      false,
      ['deriveKey']
    );
    return crypto.subtle.deriveKey(
      { name: 'PBKDF2', salt, iterations: 100_000, hash: 'SHA-256' },
      keyMaterial,
      { name: 'AES-GCM', length: 256 },
      false,
      ['encrypt', 'decrypt']
    );
  }

  async encrypt(data: string, key: CryptoKey): Promise<string> {
    const iv = crypto.getRandomValues(new Uint8Array(12));
    const ciphertext = await crypto.subtle.encrypt(
      { name: 'AES-GCM', iv },
      key,
      new TextEncoder().encode(data)
    );
    // Retourne IV + ciphertext encodés en base64
    const combined = new Uint8Array(iv.byteLength + ciphertext.byteLength);
    combined.set(iv);
    combined.set(new Uint8Array(ciphertext), iv.byteLength);
    return btoa(String.fromCharCode(...combined));
  }

  async decrypt(encoded: string, key: CryptoKey): Promise<string> {
    const combined = Uint8Array.from(atob(encoded), c => c.charCodeAt(0));
    const iv = combined.slice(0, 12);
    const ciphertext = combined.slice(12);
    const plaintext = await crypto.subtle.decrypt(
      { name: 'AES-GCM', iv },
      key,
      ciphertext
    );
    return new TextDecoder().decode(plaintext);
  }
}

4.4 Moteur de widgets et récupération REST
Le ApiFetchService est responsable de l'intégralité du cycle de vie d'une requête widget : déchiffrement des credentials, construction de la requête HTTP, exécution, extraction des valeurs via le data mapping, et normalisation vers un CleanedMetricPayload.

// core/services/api-fetch.service.ts
fetchWidgetData(
  widget: Widget,
  decryptionKey: CryptoKey
): Observable<CleanedMetricPayload> {
  return from(this.buildRequest(widget, decryptionKey)).pipe(
    switchMap(request => this.http.request(request)),
    map(response => this.applyDataMapping(response, widget.dataMapping, widget.title)),
    timeout(10_000),
    catchError(error => {
      const code = this.classifyError(error);
      return throwError(() => ({ code, widgetId: widget.id }));
    })
  );
}

private applyDataMapping(
  response: unknown,
  mapping: DataMapping,
  title: string
): CleanedMetricPayload {
  const get = (path: string) =>
    path.split('.').reduce((obj: unknown, key) =>
      obj && typeof obj === 'object' ? (obj as Record<string,unknown>)[key] : undefined,
      response
    );
  return {
    widgetTitle: title,
    primaryValue: get(mapping.primaryValuePath) as number | string,
    series: mapping.seriesPath
      ? (get(mapping.seriesPath) as Array<{label:string, value:number}>)
      : undefined
  };
}

4.5 Intégration de l'IA avec Groq
Le GroqService encapsule les appels vers le backend proxy de PulseGrid, qui seul détient la clé Groq. Il supporte le streaming SSE (Server-Sent Events) pour une expérience de rendu progressif de l'analyse, donnant à l'utilisateur le sentiment d'une réponse quasi-instantanée.

// core/services/groq.service.ts
@Injectable({ providedIn: 'root' })
export class GroqService {
  private readonly API_URL = '/api/ai/analyze';

  analyze(request: AnalysisRequest): Observable<string> {
    return new Observable<string>(observer => {
      fetch(this.API_URL, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': `Bearer ${this.authService.getAccessToken()}`
        },
        body: JSON.stringify(request)
      }).then(async response => {
        if (!response.ok) {
          if (response.status === 429) {
            observer.error({ code: 'AI_RATE_LIMITED' });
          } else {
            observer.error({ code: 'AI_UNAVAILABLE' });
          }
          return;
        }
        const reader = response.body!.getReader();
        const decoder = new TextDecoder();
        while (true) {
          const { done, value } = await reader.read();
          if (done) { observer.complete(); break; }
          observer.next(decoder.decode(value));
        }
      });
    });
  }
}

5.2 Tests unitaires
5.2.1 Tests du CryptoService
Le CryptoService est le composant le plus critique en termes de sécurité. Sa suite de tests vérifie la cohérence du schéma de dérivation de clé et du chiffrement/déchiffrement :

// crypto.service.spec.ts
describe('CryptoService', () => {
  let service: CryptoService;

  beforeEach(() => {
    TestBed.configureTestingModule({});
    service = TestBed.inject(CryptoService);
  });

  it('devrait chiffrer et déchiffrer une valeur identique', async () => {
    const salt = service.generateSalt();
    const key = await service.deriveKey('mot-de-passe-test', salt);
    const original = 'sk_live_cle_api_secrete';
    const encrypted = await service.encrypt(original, key);
    const decrypted = await service.decrypt(encrypted, key);
    expect(decrypted).toBe(original);
  });

  it('devrait retourner un ciphertext différent à chaque chiffrement (IV unique)', async () => {
    const salt = service.generateSalt();
    const key = await service.deriveKey('password', salt);
    const enc1 = await service.encrypt('data', key);
    const enc2 = await service.encrypt('data', key);
    expect(enc1).not.toBe(enc2); // IV aléatoire → ciphertext différent
  });

  it('devrait échouer avec une clé différente du mot de passe d\'origine', async () => {
    const salt = service.generateSalt();
    const key1 = await service.deriveKey('bon-mdp', salt);
    const key2 = await service.deriveKey('mauvais-mdp', salt);
    const encrypted = await service.encrypt('secret', key1);
    await expect(service.decrypt(encrypted, key2)).rejects.toThrow();
  });
});

Conclusion Générale et Perspectives
Bilan du projet
PulseGrid représente une réponse complète, cohérente et techniquement ambitieuse à une problématique métier concrète et largement répandue. En choisissant délibérément une architecture API-first sans stockage de données, le projet démontre qu'il est possible de concevoir une plateforme BI professionnelle en respectant simultanément trois contraintes a priori contradictoires : légèreté infrastructurelle, niveau de sécurité élevé et richesse fonctionnelle.

Les principaux apports techniques de ce projet sont les suivants :
•	Une architecture Angular feature-based propre, extensible et conforme aux bonnes pratiques industrielles (SOLID, lazy loading, Signals)
•	Un schéma de chiffrement client-side AES-GCM 256 bits via Web Crypto API, déployé sans aucune dépendance externe
•	Une intégration de l'IA générative (Groq + Llama 3.3 70B) avec streaming SSE, proxy sécurisé et normalisation des données avant transmission
•	Un système de widgets configurables capable de se connecter à n'importe quelle API REST avec n'importe quelle méthode d'authentification
•	Une couverture de tests unitaires de 87,3 % et 5 scénarios E2E validés

Sur le plan des compétences, ce projet a permis de mobiliser et d'approfondir des domaines complémentaires : la cryptographie applicative, la conception d'architecture logicielle, le développement frontend avancé, l'intégration de LLM dans une application web et la mise en place d'une stratégie de tests complète.

Perspectives d'évolution
PulseGrid dans sa version actuelle constitue une base solide pour de nombreuses évolutions envisagées :

Court terme (v1.1 – 3 mois)
•	Support de GraphQL et WebSocket en plus de REST
•	Alertes automatiques par email lorsqu'une métrique dépasse un seuil configuré
•	Mode collaboration : partage d'un tableau de bord en lecture seule via lien
•	Exportation automatique de rapports hebdomadaires par email

Moyen terme (v2.0 – 6 mois)
•	Application mobile native (Angular + Capacitor) en mode lecture
•	Connecteurs pré-configurés pour les API les plus courantes (Stripe, Shopify, Google Analytics, HubSpot)
•	Mode multi-utilisateurs avec gestion des rôles par projet (viewer / editor / admin)
•	White-labeling : domaine personnalisé par client pour les agences

Long terme (v3.0 – 12+ mois)
•	Intelligence prédictive : modèles de prévision des KPI basés sur l'historique (données stockées optionnellement avec consentement)
•	Marketplace de widgets communautaires : partage de configurations de widgets prêtes à l'emploi
•	API publique PulseGrid : permettre aux développeurs de créer des intégrations sur la plateforme

PulseGrid incarne la conviction que les meilleures solutions logicielles naissent d'une compréhension intime du problème que l'on cherche à résoudre. Ce projet de fin d'études a été l'occasion de vivre intégralement ce cycle, de la problématique métier à l'implémentation technique, en passant par la conception, la sécurité et la validation.
